# Sprint 1 — ReAct Loop with HO3 Underwriting Domain Logic

## Objective

Enhance the existing underwriting graph with a **bounded ReAct loop that is already aware of HO3 quoting priorities**, so the first sprint produces insurance-specific adaptive behavior instead of only generic agent orchestration.

This sprint should make the system capable of:

* identifying underwriting gaps in an HO3 submission
* prioritizing insurance-relevant next actions
* using domain-specific triggers for roof and California wildfire review
* producing a preliminary underwriting recommendation path:

  * continue
  * retrieve more evidence
  * refer
  * decline candidate
  * request missing information

---

## Enhancement approach

Do **not** replace the existing graph or decision flow.

Instead:

* preserve current nodes such as validation, enrichment, retrieval, rating, and decision
* add an insurance-aware reasoning controller on top of the current graph
* use HO3-specific rules to determine the next best step
* keep deterministic fallback behavior if reasoning is unavailable

This sprint should introduce **domain-specific action prioritization**, not just iterative execution.

---

## Sprint 1 scope

This sprint must support the following HO3 underwriting scenarios:

1. **Missing or uncertain roof information**

   * roof age missing
   * roof material missing
   * roof answers inconsistent with property facts

2. **California wildfire review**

   * property is in California
   * wildfire exposure review required
   * red-zone or high brush/hazard risk suspected

3. **Incomplete quote blocking conditions**

   * missing address quality
   * occupancy unclear
   * prior claims incomplete
   * dwelling information insufficient for quote

4. **Early referral / decline indicators**

   * high wildfire risk + aging roof
   * critical submission gaps
   * contradiction between applicant input and external evidence

---

## Insurance-aware reasoning model

The ReAct loop in Sprint 1 should reason about the quote using these domain questions:

* Is the address trustworthy enough for downstream underwriting?
* Are roof attributes complete enough for eligibility/risk review?
* Does the property require California wildfire hazard review?
* Is the submission sufficiently complete for quote progression?
* Is there already enough evidence to continue, or should the case be referred?

This gives Sprint 1 a real underwriting decision posture rather than a generic tool loop.

---

## Domain-specific action categories

The controller should choose among the following actions:

### 1. `run_existing_node`

Invoke an existing underwriting graph node such as:

* validation
* enrichment
* retrieval
* rating
* decision

### 2. `request_missing_information`

Used when quote progression is blocked by incomplete intake data.

Examples:

* missing occupancy
* missing year built
* ambiguous roof details
* incomplete applicant claims history

### 3. `flag_domain_review`

Set a domain review requirement without yet invoking advanced dynamic selection.

Examples:

* `roof_review_required`
* `wildfire_review_required`
* `guideline_review_required`

### 4. `retrieve_guideline_evidence`

Use existing retrieval/guideline lookup for underwriting support.

Examples:

* retrieve roof-age guidance
* retrieve California wildfire eligibility guidance
* retrieve refer/decline criteria

### 5. `candidate_refer`

Mark the case as requiring refer unless later evidence clears it.

### 6. `candidate_decline`

Mark the case as a decline candidate if domain conditions are severe enough, but final decline still remains subject to existing guardrails.

## Domain-specific reasoning rules

### A. Roof review triggers

Sprint 1 should flag `roof_review_required` when:

* `roof_age` is missing
* `roof_material` is missing
* `roof_age` exceeds configured threshold
* property is older and roof data is absent
* downstream quote decision materially depends on roof condition

Example initial thresholds:

```yaml
roof_age_review_threshold: 15
roof_missing_requires_review: true
```

### B. California wildfire review triggers

Sprint 1 should flag `wildfire_review_required` when:

* property state is `CA`
* address is in California and hazard review is required for quote progression
* brush/wildfire question is yes or unknown
* retrieval/guideline rules require wildfire evidence before automated progression

Example config:

```yaml
wildfire_states:
  - CA
require_wildfire_review_for_ca: true
```

### C. Submission completeness triggers

Sprint 1 should use `request_missing_information` when critical underwriting data is missing, such as:

* incomplete address
* missing occupancy
* missing year built
* missing prior claims response
* coverage data insufficient for quote path

### D. Early referral candidate triggers

Sprint 1 should mark `candidate_refer` when:

* California property with wildfire review required and missing roof information
* multiple critical gaps exist
* underwriting evidence is insufficient for safe progression
* existing graph reaches a risk decision but supporting data is incomplete

### E. Early decline candidate triggers

Sprint 1 should only produce a **decline candidate**, not a final automated decline, when:

* strong existing evidence indicates material ineligibility
* severe hazard/guideline conditions are already known
* existing system guardrails allow decline-preparation but still require final validation

## Sprint 1 reasoning flow

The loop should follow this pattern inside the current codebase:

```text
1. Read current quote state
2. Identify underwriting gaps or required reviews
3. Choose next insurance-aware action
4. Execute existing node or domain action
5. Record observation
6. Continue until:
   - enough evidence to proceed
   - refer candidate established
   - missing info blocks quote
   - max steps reached
```
